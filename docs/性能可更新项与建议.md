# 性能可更新项与建议

**更新日期**: 2026年3月  
**状态**: 已实施部分优化，其余为可选的进一步改进建议

---

## 一、本次已实施的更新

### 1. 后端缓存头扩展 ✅
- **文件**: `backend/src/middleware/performance.ts`
- **改动**: 为只读 API 增加 `/branches`（课程分支列表）的 5 分钟浏览器缓存；将可缓存路径整理为数组便于后续扩展。
- **效果**: 课程分支列表等公开数据重复访问时可直接用本地缓存，减少请求与后端压力。

### 2. 响应压缩阈值调整 ✅
- **文件**: `backend/src/middleware/performance.ts`
- **改动**: 将 `compression` 的 `threshold` 从 1024 降为 512（字节）。
- **效果**: 500B～1KB 的 JSON 响应也会被 gzip，小接口体略减小，对移动端与高延迟网络更友好。

### 3. 前端 index.html 预加载修正 ✅
- **文件**: `frontend/index.html`
- **改动**: 移除对 `/src/main.ts` 和 `/src/router/index.ts` 的预加载链接。
- **原因**: 生产构建后资源在 `dist/assets/js/` 且带 hash，原预加载路径会 404；Vite 构建时会自动注入正确的 script 与 preload。
- **效果**: 生产环境不再产生无效请求，避免控制台报错与多余连接。

## 二、建议的进一步优化（按优先级）

### 高优先级（投入小、收益明确）

| 项 | 说明 | 位置/做法 |
|----|------|-----------|
| **统一使用 executeQuery** | 部分路由直接用 `pool.query()`，未走带慢查询日志的 `executeQuery`，不利于发现慢 SQL。 | 新代码与重构时在 `backend/src/routes` 中优先使用 `executeQuery`，或在现有 `pool.query` 外包一层计时与 `DB_SLOW_QUERY_MS` 判断。 |
| **确保索引脚本已执行** | 文档中已有 `backend/scripts/mysql-indexes.sql`，需在首次部署或数据量上来后执行一次。 | 在部署文档/启动检查清单中明确写「执行 `mysql -u用户 -p 数据库 < backend/scripts/mysql-indexes.sql`」。 |
| **慢请求阈值可配置** | 慢请求日志阈值可调，便于在开发/压测时更早发现瓶颈。 | 已支持：在 `backend/.env` 中设置 `SLOW_REQUEST_MS=500`（默认 1000）。 |

### 中优先级（需一定开发量）

| 项 | 说明 | 位置/做法 |
|----|------|-----------|
| **更多只读 API 用 Redis 缓存** | 当前仅部分推荐类接口用 Redis。 | 对访问频繁、变更少的 GET（如课程列表、班级列表、知识点树）在路由内加 Redis 短 TTL 缓存（如 1～5 分钟），注意失效策略。 |
| **前端请求去重/短时缓存** | 同一接口在短时间被多次调用（如 Tab 切换、组件重复挂载）。 | 在 axios 封装或 Pinia 中对部分 GET 做请求去重或 30s 内返回缓存，减少重复请求。 |
| **图片与静态资源** | 文档中已建议 WebP、懒加载、CDN。 | 新图优先 WebP；列表/卡片图已用 `loading="lazy"` 的可保持不变；若有大量静态资源可考虑 CDN。 |

### 低优先级（长期优化）

| 项 | 说明 |
|----|------|
| **HTTP/2** | 生产环境用 Nginx/反向代理开启 HTTP/2，可减少连接与首包时间。 |
| **Service Worker / PWA** | 需要离线或“秒开”时再考虑；对当前项目非必需。 |
| **构建时 reportCompressedSize** | 当前 `reportCompressedSize: false` 有利于构建速度；若需持续监控包体积可改为 `true` 或在 CI 中单独跑一次带报告的构建。 |

## 四、环境变量速查（与性能相关）

| 变量 | 说明 | 默认 |
|------|------|------|
| `DB_SLOW_QUERY_MS` | 超过该毫秒数的 SQL 会打警告日志 | 500 |
| `DB_CONNECTION_LIMIT` | MySQL 连接池大小 | 20 |
| `REDIS_URL` | Redis 连接；未配置时部分功能降级为内存缓存 | - |
| `SLOW_REQUEST_MS` | 超过该毫秒数的 HTTP 请求会打性能警告日志 | 1000 |

## 五、后续轮次已实施的优化（继续优化项目）

### 1. 部署文档补充索引脚本 ✅
- **文件**: `README.md`、`docs/GETTING-STARTED-GUIDE.md`
- **内容**: 在「配置数据库」章节增加 2.4 执行索引脚本（`backend/scripts/mysql-indexes.sql`），并说明为推荐步骤、可提升查询性能。

### 2. 课程列表 Redis 短时缓存 ✅
- **文件**: `backend/src/routes/courses.ts`
- **内容**: GET `/api/courses` 按查询参数生成缓存键，先查 Redis/内存缓存，命中则直接返回；未命中则查库后写入缓存，TTL 120 秒（2 分钟）。依赖现有 `getCachedHotResource` / `cacheHotResource`，无 Redis 时自动走内存缓存降级。

### 3. 前端 GET 请求短时缓存 ✅
- **文件**: `frontend/src/utils/request.ts`
- **内容**: 对所有 GET 请求做 30 秒内存缓存（按 `method + url + params` 生成 key）。命中缓存则通过自定义 adapter 直接返回，不发起请求；成功响应后写入缓存。401 时清空缓存，避免登出后仍返回旧数据。

### 4. 关键路由与 executeQuery
- **结论**: `courses`、`classes` 等关键路由已使用 `executeQuery`，慢查询已纳入 `DB_SLOW_QUERY_MS` 监控，无需改动。
